The core difference

Same model. Completely different capability.

Both use the same Claude model, and Claude online with Projects is a capable assistant. The difference is everything around the model — the scale of memory, the structured layered access, the file reach and the automation I've built so it works like a trained team member rather than something I drive turn-by-turn.

🌐 Claude online (Projects)

Capable — but memory is limited and access is manual
  • Projects hold custom instructions and remember within that project's chat history
  • Memory is limited in size and scope — it can't hold the full breadth of campaigns, briefs and notes I work across
  • No direct file access — I re-upload each file every time I want it reviewed or revised
  • Brand can drift unless I restate the exact red, logo and favicon each time
  • Largely turn-by-turn — I supply the context and drive each step
  • Has the same connectors (Gmail, Notion, Drive) + skills — but I trigger each action manually, in-chat; nothing runs on its own
  • More commands and re-uploads to reach the same answer

⚙️ My Work Engine (Desktop)

Larger layered memory · direct file reach · automated
  • Reads the rules first — loads my config, brand, and manager profile before doing anything
  • Far larger, layered memory — structured access across the whole repo, not one project's history
  • Direct file access — one request reviews the latest data, past campaigns, briefs & meeting notes, no re-uploading
  • Brand-locked — exact red #E21A23, real MH logo SVG, canonical favicon, every time
  • Persistent memory — every rule learned is saved and never re-mistaken
  • Runs on a schedule — briefs, finance imports, task cascades fire on their own
  • Same connectors, but running unattended — Gmail, Notion & Drive actions fire on a schedule, repo-governed, with no one driving

2 · The layered instruction guide

How it knows which rules to apply — and when

Claude online has one layer: whatever I type. My engine has a stack of instruction layers that combine into one precise answer. Every session walks the stack top-to-bottom before producing output — global rules, then the routing tree picks the right specialist folder/skill, then those local rules are layered on top.

1

Global rules — CLAUDE.md (always loaded)

The master configuration every session inherits: my identity & role, the Manager Profile (your communication rules — lead with a number, 300-word max, solutions with every flag), the budget model, the pipeline estimation maths, KPI framework, brand quick-reference, and output rules. This is the constant baseline.

▼
2

Memory check — memory/MEMORY.md (read before any work)

Before acting, it reads the memory index: hard rules learned from past feedback, current project state, and reference pointers. Repo memory always wins over anything stale in cached context. This is what stops repeated mistakes.

▼
3

Domain Decision Tree — routes the task to the right place

A lookup table maps the topic to the exact files to read and rules to apply. e.g. BTS → read brand/campaigns/bts.md · landing page → read brand/favicon.html + brand rules · Hammad brief → read the locked template + last 2 briefs. It only reads what's relevant — not the whole repo.

▼
4

Specialist skills — auto-trigger by phrase

Domain skills fire automatically: performance-review, UTM-validator, paid-media-optimizer, personal-assistant (briefs), and brand-guidelines enforcement. Each carries its own deep procedure and rules.

▼
✓

Combine & respond — one precise, on-brand output

Global rules + memory + the domain-specific files + the skill procedure are merged into a single chain of reasoning. The result: an answer that already obeys your format, uses the right data, is on-brand, and avoids every known past mistake — without me re-explaining anything.

6 · Ongoing instruction improvement

It gets better every time I correct it

When I give feedback or a rule changes, it doesn't just fix the one output — it writes the lesson into the right memory layer so the mistake never recurs. The engine compounds: every correction permanently raises the floor.

1

I give feedback

A correction, a new constraint, or "that's not on-brand."

→
2

→
3

Save to the right layer

Written to memory/, the decision tree, or config — wherever it belongs.

→
4

Applied forever

Loaded automatically on every future session. Never re-mistaken.

Real example — the favicon rule
"Can't we simply use the exact same favicon as the US site instead of trying to recreate it?"
After three failed recreation attempts, the lesson was captured and locked as v4 into memory/feedback_favicon_use_svg_not_ico.md with a hard verification step (grep ≥ 2). Every HTML output since — including this very page — automatically points the browser at the canonical mheducation.com favicon. The mistake is now structurally impossible to repeat.
